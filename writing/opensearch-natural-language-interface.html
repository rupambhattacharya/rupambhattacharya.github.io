Building a Natural-Language Interface to OpenSearch with Claude Tool Use

July 2026 14 min read AI · AWS · OpenSearch

At ProSieben we run a production OpenSearch cluster that stores ATV (connected-TV) streaming analytics — device sessions, viewing patterns, ad break events, consent logs. The data is rich. The access pattern, until recently, was not: a product manager with a question had to file a Jira ticket, a developer had to translate it into Query DSL, and both parties had to wait for each other. Every ad-hoc question was a small tax on engineering time.

I wanted to fix that. What I built is an agent that lets anyone on the team type a question in plain English — "how many unique devices were active yesterday, broken down by manufacturer?" — and get an answer in seconds, without touching DSL or asking an engineer. This post explains how I built it, what the architecture looks like, and the specific decisions that made the security model actually hold up.

The core problem

Why this is harder than it looks

Connecting an LLM to a database sounds simple until you think about it carefully. The LLM doesn't know which indices exist. It doesn't know field names or types. And even once you tell it, it has to generate valid Query DSL — a syntax that's expressive but unforgiving. A wrong field name or a missing aggregation key doesn't just return bad data; it returns a 400 error.

Claude's tool use solves the knowledge problem elegantly. Instead of trying to stuff the entire schema into a system prompt (it's too large, and it changes), I give Claude three tools and let it discover what it needs at runtime:

  • list_indices — discover what indices exist, their document counts and sizes
  • get_mapping — fetch field types and structure for a specific index
  • search_documents — execute a Query DSL query or aggregation

The agent's first move for an unfamiliar question is almost always to call list_indices, then get_mapping on the relevant index, and only then construct a query. This is exactly how an experienced engineer approaches an unfamiliar cluster — and it means the agent is always working from current schema, not a stale snapshot I fed it weeks ago.

Architecture

Everything in one Lambda

The architecture is deliberately minimal: a single AWS Lambda function behind a Function URL. No API Gateway, no separate frontend hosting, no orchestration service. The Lambda does three things depending on the request:

  • GET request: serves the web chat UI (HTML/CSS/JS inlined in a TypeScript string)
  • POST request: runs the tool-calling loop — calls Claude, executes tools against OpenSearch, iterates until Claude has a final answer, returns it
  • Session management: keeps an in-memory conversation store so follow-up questions work ("filter that to Samsung only")
Browser (GET) ──────────────────────▶ Lambda (Function URL) │ │ serves inline chat UI ▼ Browser (POST message) ──────────────▶ Lambda │ ┌────────┴────────┐ │ │ Claude API OpenSearch (via LLM Gateway) (SigV4 signed) │ │ └────────┬────────┘ │ tool loop (up to 10 iters) │ final answer │ ▼ Browser ◀──────── JSON response

The Lambda is Node.js 24.x, 512 MB memory, 600 second timeout. The generous timeout matters: a complex multi-step query — "list indices, fetch mapping, run aggregation, run a follow-up aggregation" — can take 30–45 seconds even on a fast cluster, and most of that is waiting on Claude's API calls.

The tool-calling loop

The core of the Lambda is a simple loop: call Claude, check if the response contains a tool_use block, execute the tool, append a tool_result to the conversation, call Claude again. Repeat until Claude returns a stop_reason of end_turn with no pending tool calls. 

let messages = [...sessionHistory, { role: 'user', content: userMessage }];
let iterations = 0;

while (iterations < MAX_TOOL_ITERATIONS) {
  const response = await claude.messages.create({
    model: 'claude-sonnet-4-6',
    max_tokens: 4096,
    system: SYSTEM_PROMPT,
    tools: TOOL_DEFINITIONS,
    messages,
  });

  if (response.stop_reason === 'end_turn') {
    return extractText(response);
  }

  // Claude wants to call a tool
  const toolUse = response.content.find(b => b.type === 'tool_use');
  const toolResult = await executeTool(toolUse.name, toolUse.input);

  messages = [
    ...messages,
    { role: 'assistant', content: response.content },
    { role: 'user', content: [{ type: 'tool_result', tool_use_id: toolUse.id, content: toolResult }] },
  ];

  iterations++;
}

The loop cap of 10 iterations is a practical guardrail. In production, simple questions resolve in 2–3 iterations. Complex multi-step explorations top out at 6–7. I've never seen a legitimate query need more than that.

Connecting to OpenSearch: SigV4 and IAM

OpenSearch on AWS uses SigV4 request signing — the same signing mechanism as every other AWS API. My Lambda has an IAM execution role, and I use the @opensearch-project/opensearch client with the aws-sdk credentials provider to sign every request automatically: 

import { Client } from '@opensearch-project/opensearch';
import { AwsSigv4Signer } from '@opensearch-project/opensearch/aws';
import { defaultProvider } from '@aws-sdk/credential-provider-node';

const client = new Client({
  ...AwsSigv4Signer({
    region: 'eu-west-1',
    service: 'es',
    getCredentials: defaultProvider(),
  }),
  node: process.env.OPENSEARCH_ENDPOINT,
});

This means there are no credentials to manage or rotate. The Lambda's IAM role is the credential. When the role changes (on rotation, on Lambda update), the signing just works.

Security: defence in depth

Why read-only access is harder to guarantee than it sounds

The first time I described this project, someone asked: "couldn't the LLM just decide to delete something?" It's a fair question. The honest answer is: the system prompt alone is not enough. I needed security controls at every layer independently, so that no single failure — a prompt injection, a bug in my loop, a future model version that reasons differently — could result in a write.

Here's the full stack:

LayerControlWhat it prevents
Tool definitions Only search_documents, list_indices, get_mapping exist No code path for write operations; Claude cannot call a tool that doesn't exist
IAM policy es:ESHttpGet and es:ESHttpPost only No ESHttpPut, ESHttpDelete, ESHttpPatch at the AWS level
FGAC role ai_agent_readonly in OpenSearch Security Even with HTTP POST (needed for _search), no write/delete/admin at the index level
Result cap size capped at 100 documents Prevents accidental bulk data exfiltration via large queries
System prompt Claude instructed it has read-only access First line of defence; not the only one
The key insight: HTTP POST is required for OpenSearch _search — search requests with complex DSL bodies go via POST, not GET. This means IAM-level read-only cannot simply block all POSTs. The FGAC layer exists precisely to enforce read-only semantics at the index level, independently of the HTTP method.

Setting up FGAC (Fine-Grained Access Control)

FGAC is OpenSearch's internal security layer. It sits below IAM: IAM controls whether the AWS API call reaches OpenSearch at all; FGAC controls what that request is allowed to do once it arrives.

I created a custom role called ai_agent_readonly and mapped the Lambda's IAM role ARN to it. The role grants exactly what the agent needs and nothing more: 

PUT _plugins/_security/api/roles/ai_agent_readonly
{
  "cluster_permissions": ["cluster_composite_ops_ro"],
  "index_permissions": [{
    "index_patterns": ["*"],
    "allowed_actions": [
      "read",
      "indices:data/read/*",
      "indices:admin/mappings/get",
      "indices:monitor/stats"
    ]
  }]
}
PUT _plugins/_security/api/rolesmapping/ai_agent_readonly
{
  "backend_roles": [
    "arn:aws:iam::<account-id>:role/atv-data-ai-agent-lambda-role-prod"
  ]
}

The role mapping uses the Lambda's IAM role ARN as a backend role. OpenSearch extracts the role from the SigV4-signed request and applies the FGAC policy. No username, no password, no API key to manage — the IAM identity is the identity.

The LLM Gateway and key management

We run a company-managed LLM gateway (a LiteLLM proxy) that handles model routing, rate limiting, and audit logging across all teams. Rather than hardcoding an Anthropic API key, I call the gateway with a virtual key stored in SSM Parameter Store as a SecureString: 

const apiKey = await ssm.getParameter({
  Name: '/atv-data/prod/claude-api-key',
  WithDecryption: true,
}).promise();

const claude = new Anthropic({
  apiKey: apiKey.Parameter.Value,
  baseURL: process.env.LLM_GATEWAY_URL,
});

The Lambda's IAM role has ssm:GetParameter on that specific parameter path — nothing broader. The key is never in source code, environment variables at rest, or logs. The gateway provides an additional layer of rate limiting and visibility into which teams are spending what.

The conversation experience

Multi-turn state

One of the things that makes this genuinely useful rather than just a novelty is that it remembers the conversation. A PM can ask "show me sessions by manufacturer this week", then "filter to just Samsung", then "what's the trend for Samsung over the last month?" — each question building on the last, without repeating context.

State is kept in a simple in-memory map keyed by session ID (a UUID generated on first request). The Lambda returns the session ID in the response; subsequent requests include it. There's a 30-minute TTL; if the Lambda cold-starts, sessions are lost, but in practice that's rare during working hours.

For most use cases this is good enough. The obvious upgrade — if session persistence across cold starts matters — is to move the session store to DynamoDB.

What the tool loop looks like in practice

Take the question: "How many device sessions happened yesterday?" Here's what actually happens inside the loop:

1

Claude receives the question along with the system prompt and tool definitions. It decides to call list_indices to discover what's available.

2

Lambda calls list_indices. Returns index names, doc counts, and sizes. Claude sees device-data-session-* indices and identifies them as relevant.

3

Claude calls get_mapping on device-data-session-2026.06.25 to confirm which field holds the timestamp. Finds @timestamp.

4

Claude calls search_documents with a range query on @timestamp for yesterday, and a value_count aggregation on a unique session field. Lambda signs and executes the request against OpenSearch.

5

Claude synthesises the answer: "There were 1,247,832 device sessions yesterday. This is 3.2% higher than the same day last week." The loop ends.

The whole exchange takes about 15–25 seconds. Most of that is latency across two round-trips to an external API and one OpenSearch query.

What makes this different from a chatbot over docs

A lot of "AI + data" demos work by pre-indexing documents and doing RAG. This is different in a few ways that matter for production analytics:

It queries live data. The agent hits the actual OpenSearch cluster on every request. If data was ingested five minutes ago, it's queryable. There's no indexing lag, no stale cache.

It uses aggregations, not document retrieval. The interesting analytics questions aren't "show me document 42" — they're "how many unique devices per day for the last week, broken down by platform?" That's a date_histogram aggregation with a cardinality sub-aggregation. Claude generates valid aggregation DSL and interprets the bucket results correctly.

The schema is dynamic. Index mappings change over time. Tool-based discovery means the agent always works with the current schema, not one I froze into a prompt last month.

Things that surprised me

Claude is better at Query DSL than I expected

I was prepared to spend time prompt-engineering around DSL mistakes. In practice, Claude's Query DSL is solid. The main failure mode I encountered was on complex nested aggregations where the index mapping had unusual field types. The fix was to always call get_mapping first and include the relevant field types in the tool result — once Claude knows a field is keyword vs text, it makes the right aggregation choice.

Error handling is part of the intelligence

When an OpenSearch query fails — wrong field name, index not found, query parse error — I return the error as a tool result rather than throwing an exception. Claude reads the error, adjusts, and retries with a corrected query. This self-correction behaviour is one of the most useful properties of tool-based agents. The system degrades gracefully rather than hard-failing.

The result size cap matters more than you'd think

Without a hard size cap, I noticed Claude would sometimes request 1,000 or 10,000 documents on exploratory queries. Pulling that into the context is slow, expensive, and rarely useful — aggregations answer the real question much better. Enforcing MAX_SIZE=100 server-side (not just in the prompt) forces aggregation-first behaviour and keeps response times sensible.

What's next

The agent is deployed and in daily use by product and engineering. A few directions I'm thinking about:

  • Scheduled reports via EventBridge. Trigger the Lambda daily with a prompt like "summarise yesterday's KPIs" and post results to a Teams channel. The backend is already the right shape for this.
  • Anomaly alerting. A periodic check: "are any indices showing document counts more than 2 standard deviations below their 7-day average?" The agent can do the math; I just need to hook it up to a notification channel.
  • Multi-source tools. Adding CloudWatch metrics alongside OpenSearch would let Claude correlate Lambda errors with ingestion gaps — the kind of debugging question that currently requires jumping between dashboards.

The core pattern — give an LLM a small set of well-defined tools, enforce security at every layer independently, keep the infrastructure minimal — scales beyond OpenSearch. I'm using a similar approach for the GenAI developer CLI we run internally, and the same principles hold.
